# Longevity Research — Knowledge Tree

A markdown-source knowledge base of computational entry points into
longevity and reverse-aging research, written for a non-biologist
computer-scientist contributor.

**Design pattern:** plain markdown with YAML front matter + `[[wikilinks]]`
→ build step produces a TF-IDF + graph index → a linter keeps it
consistent. No database, no server, no query CLI — the wiki *is* the
interface.

**Current coverage:** 75 curated sources, 54 topics, 2 analyses
(tier ranking + evidence-tier rubric), 132 wiki pages, ~1100
wikilink edges.

## Where to start reading

If you came here from a Reddit post or a link, read in this order:

1. [`recommendations.md`](recommendations.md) — actionable summary,
   split into "Interventions to do" (with evidence-tier tags) and
   "Biomarkers to measure" (with target ranges, the intervention
   that moves each, and a marker-vs-cause caveat).
2. [`wiki/analysis/evidence-tiers.md`](wiki/analysis/evidence-tiers.md)
   — the T0-T7 maturity ladder used to rate every intervention.
3. [`wiki/analysis/promising-reverse-aging.md`](wiki/analysis/promising-reverse-aging.md)
   — tier-ranked frontier of partial reprogramming, gene therapies,
   senolytics, foundation-model clocks.
4. [`wiki/topics/`](wiki/topics/) for depth on a specific area.

This is a research wiki, not medical advice. Discuss anything in the
medical-optimization or off-label tiers with a physician.

## Contributing via AI agent

The wiki is designed to be maintained by an LLM. Open `claude` (or
cursor / codex / aider) in a directory you're happy for it to touch
and prompt:

**Clone:**

> Clone <https://github.com/clankwright/science.git> here. Then read
> biology/longevity/CLAUDE.md so you know how the wiki works.

**Add a paper:**

> Ingest this paper into the longevity wiki: <PMC or DOI URL>.
> Follow biology/longevity/CLAUDE.md.

**Add several papers / topic sweep:**

> Research and ingest the strongest recent papers on <topic, e.g.
> "GLP-1 agonists and biological aging"> into the longevity wiki.
> Follow biology/longevity/CLAUDE.md.

**Update the analysis after new ingests:**

> Update biology/longevity/wiki/analysis/promising-reverse-aging.md
> to integrate the new papers. Lint clean.

**Update the reader-facing recommendations doc:**

> Update biology/longevity/recommendations.md to cite any new
> supporting evidence and re-tier where warranted.

The agent handles `sources.json`, license maps, download/convert,
paper pages, topic links, lint, and log entries. You handle the PR.

**Open the PR:**

> Commit (no AI attribution per biology/longevity/CLAUDE.md) and
> prep a PR branch.

Then push and open the PR on GitHub if the agent doesn't have push
permissions.

## Layout

```
biology/longevity/
├── sources.json          # canonical 40-source manifest
├── LICENSES.md           # generated; per-source license table
├── sources/              # raw downloads (gitignored)
│   ├── html/
│   ├── pdf/
│   └── md/               # converted markdown
├── wiki/
│   ├── index.md          # root, topic taxonomy, project leads
│   ├── analysis/*.md     # tier rankings + syntheses
│   ├── papers/*.md       # one page per source
│   ├── topics/*.md       # one page per research topic
│   └── build/            # generated by scripts/index.py (gitignored)
└── scripts/
    ├── _wiki.py          # shared front-matter + wikilink parsing
    ├── download.py       # fetch HTML/PDF for every source
    ├── convert.py        # HTML → md (trafilatura) / PDF → md (pdftotext)
    ├── index.py          # TF-IDF + wikilink graph + keyword index
    ├── lint.py           # link / metadata / parity / license checks
    └── licenses.py       # apply license fields + regenerate LICENSES.md
```

## Usage

All scripts run from the `biology/longevity/` directory. Network-free steps
first; network-dependent steps last.

```bash
cd biology/longevity

# (1) Build the index over the wiki — no network needed.
python3 scripts/index.py

# (2) Lint the wiki + manifest — no network needed.
python3 scripts/lint.py

# (3) Refresh per-source license tags + regenerate LICENSES.md.
#     Edit LICENSE_MAP in scripts/licenses.py first when adding sources.
python3 scripts/licenses.py all

# (4) Download primary sources (HTML / PDF). Needs unrestricted egress
#     to arXiv, bioRxiv, medRxiv, PMC, Nature, MDPI, etc. Re-runnable;
#     skips already-cached files.
python3 scripts/download.py

# (5) Convert downloaded HTML / PDF to clean markdown under sources/md/.
python3 scripts/convert.py
```

A typical workflow loop after editing wiki pages or `sources.json`:

```bash
python3 scripts/licenses.py all && python3 scripts/index.py && python3 scripts/lint.py
```

## Reading the wiki

- [`wiki/analysis/promising-reverse-aging.md`](wiki/analysis/promising-reverse-aging.md)

## Build artifacts (`wiki/build/`)

| file | contents |

## Linter checks

`scripts/lint.py` exits with the number of errors found.

1. Broken `[[wikilinks]]`
2. Paper pages missing `url:` front matter
3. Paper pages missing `topics:` front matter
4. Orphan topic pages (no inbound links)
5. Papers not referenced by any other page
6. `sources.json` ↔ `wiki/papers/*.md` id parity
7. Front-matter URL-valued fields that aren't actually URLs
8. Every `sources.json` entry has a `license:` field

## Licensing

Every source in `sources.json` carries a `license:` (and usually a
`license_note:`) reflecting a best-effort lookup of the venue's
standard license. `scripts/licenses.py` is the source of truth:

```bash
python3 scripts/licenses.py apply      # write license fields into sources.json
python3 scripts/licenses.py generate   # rebuild LICENSES.md
python3 scripts/licenses.py all        # both
```

`LICENSES.md` (regenerated in both `biology/` and inside
`biology/longevity/`) groups the 40 sources into:

- **Redistributable** — CC-BY-4.0 / CC-BY-3.0 / public-domain
- **Restricted** — CC-BY-NC etc. (currently empty)
- **Varies per paper** — preprints with author-selectable license
- **Not redistributable** — all-rights-reserved or website-ToS

If you commit converted full-text under `sources/md/`, restrict the
commit to redistributable sources and include the license + DOI in
the YAML front matter of each committed file. `sources/` is in
`.gitignore` by default so you don't do this accidentally.

## Sandbox / network restrictions

`scripts/download.py` requires direct HTTP access to many academic
hosts. Some execution environments enforce an egress firewall that
returns `403 host_not_allowed` for arXiv / bioRxiv / medRxiv / PMC /
Nature / MDPI / etc. In those cases, run the download step on an
unrestricted machine and rsync `sources/` over.

The initial seed of `wiki/papers/*.md` was populated from
search-engine summaries rather than primary full text; running
`download.py` + `convert.py` is what replaces the stubs with real
content under `sources/md/`.

## Adding a new source

1. Append an entry to `sources.json` with a stable `id` slug and the
   best open-access URLs (`pdf_url` and `alt_url` may be lists).
2. Add an entry to `LICENSE_MAP` in `scripts/licenses.py`.
3. Create `wiki/papers/<id>.md` with YAML front matter (`id`, `title`,
   `url`, `year`, `venue`, `access`, `kind`, `topics`).
4. Link it from the relevant `wiki/topics/*.md` pages.
5. Run the lint loop:

```bash
python3 scripts/licenses.py all && python3 scripts/index.py && python3 scripts/lint.py
```

## Adding a new topic

1. Create `wiki/topics/<slug>.md` with YAML `topic: <slug>` front matter.
2. Link to and from the relevant paper pages.
3. Re-run `scripts/index.py` + `scripts/lint.py`.
